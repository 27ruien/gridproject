# Kiviflow 产品化重构 PRD

## 1. 产品定位与目标用户

Kiviflow 是面向中小型产品研发团队、客户交付团队和项目型组织的轻量项目管理平台前端雏形。它优先解决“项目如何启动、如何看清进度、如何推动事项、如何沉淀交付节奏”的问题，而不是一开始追求复杂协同、权限和集成。

目标用户：
- 项目负责人：需要快速创建项目、判断风险、推动下一步。
- 产品/研发/测试成员：需要明确自己负责的事项、优先级、状态和截止日期。
- 客户交付 PM：需要阶段、里程碑、交付物、风险和验收口径。
- 团队管理者：需要跨项目看健康度、未完成事项和高风险事项。

## 2. 当前为什么不商业化

现状已经有模板、项目和事项雏形，但商业化感不足主要来自：
- 业务模型薄：事项只有基础字段，缺少负责人、截止日期、优先级、描述、类型与项目上下文的一致表达。
- 模板弱：模板只是影响页面展示，未充分决定默认视图、事项类型、字段、空状态、工作流和引导。
- 项目空间弱：用户进入项目后不能一眼判断项目进度、风险、下一步重点。
- 架构弱：localStorage 与业务逻辑直接混在页面组件，未来接 API、权限、协作会很痛。
- 视觉与信息密度不足：像 demo 而不是项目管理工具，关键操作反馈、空状态、列表扫描效率不足。

## 3. 商业化项目管理工具的核心体验

- 快速启动：选择模板即可获得默认流程、视图、事项类型、字段和示例结构。
- 可扫描：工作台、项目列表、事项列表、看板列、阶段视图都能快速看出重点。
- 可推进：事项状态流转清晰，每次操作有反馈。
- 可判断：项目空间顶部直接展示进度、风险、未完成事项、下一步。
- 可扩展：存储、服务、领域模型与视图解耦，为后端 API、登录权限、飞书集成、协作活动流预留边界。

## 4. 核心对象模型

- Workspace：组织空间。当前默认单空间，未来承载成员、权限、应用集成与计费。
- Project：项目。包含模板、负责人、健康度、状态、时间范围和项目目标。
- Template：项目模板。决定默认视图、事项类型、字段、工作流、引导文案和空状态。
- Issue：事项。包含类型、优先级、负责人、截止日期、描述、状态、所属项目、下一步和活动。
- Workflow：流程。当前统一为“未开始、进行中、已完成、已验收”，未来可由模板扩展。
- View：项目视图。敏捷默认概览、Backlog、Sprint、看板；瀑布默认概览、阶段、里程碑、交付物、风险、验收。
- Milestone：项目级里程碑。由模板初始化到项目实例，支持状态维护、截止日期和交付焦点。
- Activity：活动。当前记录状态变更与创建动作的本地事件，未来映射协作动态。

## 5. 模板差异

敏捷研发：
- 默认视图：概览、Backlog、Sprint、看板、版本、复盘。
- 事项类型：Epic、需求、任务、缺陷、技术债。
- 核心关注：优先级、Sprint 计划、吞吐、缺陷闭环。
- 空状态引导：先建立需求池，再选择 P0/P1 进入 Sprint。

瀑布交付：
- 默认视图：概览、阶段计划、里程碑、交付物、风险、验收。
- 事项类型：阶段、任务、交付物、风险、变更、验收项。
- 核心关注：阶段门、交付物、风险、客户验收。
- 空状态引导：先确认范围和阶段，再补里程碑与验收项。

## 6. 第一版不做什么

- 不做后端、登录、权限、飞书集成、多人实时协同。
- 不做复杂甘特图、自动排期、依赖网络图。
- 不做自定义模板编辑器，只保留模板模型扩展点。
- 不做复杂报表配置，只保留关键指标。

## 7. P0/P1/P2 范围

P0：
- 项目创建可用，模板选择明确。
- 敏捷和瀑布项目空间可用。
- 事项创建、编辑、状态流转、localStorage 持久化可用。
- 构建通过，核心页面可运行。

P1：
- Dashboard 有商业化工作台感。
- 项目空间有进度、风险、下一步摘要。
- 事项字段完整：类型、优先级、负责人、截止日期、描述、状态、所属项目。
- localStorage 抽象为 storage adapter，业务层不直接依赖浏览器存储。
- 业务逻辑进入 domain/services/composables。

P2：
- 活动流雏形。
- 空状态、错误状态、加载状态。
- 移动端基础适配。
- 轻量领域逻辑测试。

## 8. 验收标准

- `npm install` 成功。
- `npm run build` 成功。
- 工作台、项目库、工时填报、项目空间可运行。
- 可创建敏捷项目和瀑布项目。
- 可创建、编辑事项，并推进状态。
- 刷新后数据仍在。
- UI 明显不像临时 demo。
- 无 P0 问题；P1 已修复或有合理说明。
